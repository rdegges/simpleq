# SimpleQ Quality Roadmap

This roadmap turns the current codebase review into an execution plan focused on
one outcome: when Python developers inspect SimpleQ, they should immediately see
a library that is reliable, well-tested, intentionally designed, and safe to
adopt in production.

## Baseline

The repository already has a strong foundation:

- `uv run ruff check .` passes
- `uv run mypy simpleq` passes
- `uv run pytest -q -m "not live"` passes with `354 passed, 16 deselected`

The clearest quality gap today is coverage honesty:

- `uv run pytest -m "not live" --cov=simpleq --cov-report=term-missing --cov-fail-under=97`
  currently fails at about `92.8%`
- Biggest uncovered modules are `simpleq/cli.py` and `simpleq/testing.py`

## North Star

By the end of this roadmap, SimpleQ should have:

- a green quality gate that matches the repository's stated standard
- executable docs and examples in CI
- smaller, easier-to-review modules at public entrypoints
- stronger typing at core boundaries
- better resilience coverage for real SQS failure modes
- clearer release and compatibility signals for the community

## Success Metrics

- Coverage gate passes in CI without caveats
- Overall coverage is `>= 95%`
- `simpleq/cli.py` and `simpleq/testing.py` are each `>= 90%` covered
- Docs build stays green and code examples are exercised automatically
- Public core modules expose fewer `Any`-typed seams
- Release process includes changelog and API stability notes

## Guiding Rules

- Keep each roadmap item PR-sized
- Prefer tests before refactors
- Do not mix feature work with structural cleanup in the same PR
- Ship improvements that increase user trust before improvements that are mostly internal

## Phase 1: Close the Trust Gaps

Target: 1 to 2 weeks

### PR 1: Make the coverage bar real

Scope:

- Add missing tests for the highest-miss paths in `simpleq/cli.py`
- Add missing tests for the highest-miss paths in `simpleq/testing.py`
- Add targeted branch tests for `simpleq/worker.py`, `simpleq/queue.py`, and `simpleq/sqs.py`
- Keep the `97%` gate only if it can be reached quickly and honestly
- If `97%` is still unrealistic after targeted work, temporarily lower the gate to `95%`
  and create a follow-up issue to raise it again

Acceptance criteria:

- `uv run pytest -m "not live" --cov=simpleq --cov-fail-under=<chosen-threshold>` passes
- No module in the main execution path is below `90%`
- Coverage threshold in CI matches the level the repo can actually sustain

Why this matters:

- A failing quality bar erodes confidence fast for contributors and evaluators

### PR 2: Make docs and examples executable

Scope:

- Add smoke tests for `examples/basic.py`
- Add tests for the scaffold generated by `simpleq init`
- Add a lightweight README verification path for the quick-start flow
- Ensure the most important examples run against `InMemoryTransport` or LocalStack in CI

Acceptance criteria:

- The README quick-start path is exercised automatically
- At least the main example and generated scaffold are tested in CI
- Docs changes that break examples fail before release

Why this matters:

- Libraries feel trustworthy when their docs are known-good, not aspirational

## Phase 2: Improve Maintainability at the Edges

Target: 1 to 2 weeks

### PR 3: Split the CLI into focused modules

Scope:

- Break `simpleq/cli.py` into smaller command modules
- Keep Typer wiring shallow in the root module
- Move parsing, import loading, worker reload logic, and command handlers into separate units
- Preserve the current command surface exactly

Suggested structure:

- `simpleq/cli/__init__.py`
- `simpleq/cli/worker.py`
- `simpleq/cli/queue.py`
- `simpleq/cli/task.py`
- `simpleq/cli/job.py`
- `simpleq/cli/shared.py`

Acceptance criteria:

- CLI behavior is unchanged from a user perspective
- Each CLI module is easy to test in isolation
- The old monolithic file is gone or drastically reduced

Why this matters:

- A smaller surface area makes outside contributions much less intimidating

### PR 4: Replace `Any`-heavy seams with protocols

Scope:

- Introduce typed protocols for transport, queue-like behavior, metrics, and logger usage
- Tighten public signatures in `simpleq/client.py`, `simpleq/task.py`, `simpleq/worker.py`, and `simpleq/sqs.py`
- Reserve `Any` for truly dynamic payloads, not infrastructure boundaries

Acceptance criteria:

- Core runtime modules have materially fewer `Any`-typed parameters and returns
- Mypy still passes in strict mode
- Test doubles can implement explicit protocols instead of ad hoc duck typing

Why this matters:

- Strong types communicate design maturity to Python users immediately

## Phase 3: Reliability and Behavioral Confidence

Target: 2 weeks

### PR 5: Add failure-mode coverage for real SQS behavior

Scope:

- Add tests for partial batch failures
- Add tests for malformed AWS responses
- Add tests for stale queue URLs and cache invalidation edge cases
- Add tests for ack failures, heartbeat failures, and receive timeouts
- Add tests for retry behavior under jitter and non-retryable errors

Acceptance criteria:

- Critical failure paths are covered by explicit regression tests
- Failures produce clear metrics and structured log events
- Worker behavior under failure is documented and stable

Why this matters:

- Reliability is the feature people are actually buying when they adopt a queue

### PR 6: Treat `InMemoryTransport` as a contract, not just a helper

Scope:

- Define which SQS behaviors `InMemoryTransport` intentionally mimics
- Add parity tests that run the same scenarios against LocalStack and `InMemoryTransport`
- Document unsupported behaviors clearly instead of leaving them implicit

Acceptance criteria:

- Developers know exactly when `InMemoryTransport` is sufficient
- The test transport does not drift silently from expected queue semantics

Why this matters:

- Local confidence drives adoption, and test transports shape first impressions

## Phase 4: Community Trust Signals

Target: 1 week

### PR 7: Publish maintenance guarantees

Scope:

- Add `CHANGELOG.md`
- Add a short API stability and deprecation policy
- Document supported Python and AWS usage expectations clearly
- Clarify what is considered public API versus internal implementation

Acceptance criteria:

- Every release has visible user-facing notes
- Contributors can tell what is safe to depend on
- Breaking changes have a predictable communication path

Why this matters:

- Popular libraries are not just good; they are legible and dependable

### PR 8: Tighten contributor and release discipline

Scope:

- Add a PR checklist that includes docs smoke tests and coverage expectations
- Add a release checklist that requires changelog updates and docs verification
- Add issue labels or milestones for quality work so this effort stays visible

Acceptance criteria:

- Quality work is visible in the normal development process
- Releases no longer rely on maintainer memory

Why this matters:

- Repetition and consistency are what make quality durable

## Recommended Order

1. PR 1: coverage bar and missing tests
2. PR 2: executable docs and examples
3. PR 3: CLI split
4. PR 4: typed protocols
5. PR 5: failure-mode coverage
6. PR 6: `InMemoryTransport` contract tests
7. PR 7: changelog and stability policy
8. PR 8: contributor and release discipline

## What Not to Do

- Do not start with a giant refactor
- Do not raise the quality bar in docs without making CI enforce it
- Do not chase 100% coverage if it causes low-value tests
- Do not mix product roadmap items with quality roadmap items

## Suggested Cadence

- Week 1: PR 1
- Week 2: PR 2
- Week 3: PR 3
- Week 4: PR 4
- Weeks 5 and 6: PR 5 and PR 6
- Week 7: PR 7 and PR 8

## Definition of Done

This roadmap is complete when a new engineer can clone the repo, run the
documented commands, read the main modules, and come away with the same
conclusion: "this library is serious, thoughtfully maintained, and safe to try."
